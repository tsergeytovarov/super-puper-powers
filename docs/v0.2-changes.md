# SPP v0.2 — план изменений по итогам догфуда

> Источник: два независимых догфуд-прогона (tg-бот-конвертер, web-сплитер счёта),
> оба прошли pipeline 0–9. Замечания в `NOTES.md` каждого прогона.
> Решения по объёму: правим раздел A (подтверждено обоими) + важные уникальные из B;
> git-деградацию — точечно (не полный dry-run режим); фазы 8–9 — под-режим strategy-only.

## Легенда происхождения (важно для vendored)

- **new** — оригинальный SPP-скилл, правим свободно.
- **reworked** — переработка upstream (spec-writing, plan-writing, using-super-puper-powers); правим свободно, шапка уже `modified`.
- **VENDORED** — копия upstream as-is-с-минимальными-правками; любая правка = модификация,
  обновить строку `Modifications` в шапке скилла И запись в `UPSTREAM.md`. Правки минимальны.

## Группа A — подтверждено обоими прогонами

### A1. Доменный чеклист обязательных решений в spec-writing
- **Файл:** `skills/spec-writing/SKILL.md` (reworked)
- **Проблема:** автор systematically недоспецифицирует формат денег/округление/копирайт
  состояний/инварианты — ровно то, что потом ловит spec-review за 5–6 раундов.
- **Правка:** добавить в Process секцию «Domain decisions checklist» — перед self-review
  агент обязан явно зафиксировать в спеке решения по типовым осям, если они применимы:
  деньги → формат/округление/разделитель; списки → сортировка/пустое состояние;
  ввод → парсинг и валидация каждого поля; числа с плавающей точкой → инвариант в целых;
  тексты состояний → точный копирайт всех веток (успех/ошибка/пусто).
- **Критерий:** секция присутствует; self-review-шаг ссылается на неё.
- **Ожидаемый эффект:** меньше раундов spec-review (корень переусердствования).

### A2. Свести авторский self-review к скану, не к ловле логики
- **Файлы:** `skills/spec-writing/SKILL.md`, `skills/plan-writing/SKILL.md` (reworked)
- **Проблема:** авторский self-review в обоих прогонах поймал 0 реальных дефектов; всё
  вытащил только внешний свежий контекст.
- **Правка:** переформулировать self-review-шаг: это скан на плейсхолдеры + покрытие
  доменного чеклиста (A1) + полнота относительно must-сценариев, БЕЗ претензии на ловлю
  логических дефектов — их ловит внешний spec-review/plan-review. Явно сказать: «не полагайся
  на self-review как на замену внешнему ревью».
- **Критерий:** self-review-шаг переформулирован в обоих; убрана иллюзия, что он ловит логику.

### A3. Lite-профиль церемонии для крошечных идей
- **Файлы:** `skills/using-super-puper-powers/SKILL.md` (reworked, основное решение здесь);
  `skills/plan-writing/SKILL.md` (reworked, оценка размера).
- **Проблема:** 10 фаз, ~10 субагентов, 10 артефактов на ~100 строк кода. SDD-машинерия
  (worktree, task-brief, 2 ревью на задачу) — гигантский оверхед для 2–3 крошечных задач.
- **Правка (НЕ трогая vendored SDD глубоко):** ввести в оркестратор понятие `pipeline_profile`
  (full | lite) в state. plan-writing оценивает размер плана; если план ≤ 3 задач И суммарно
  ≤ ~150 строк — рекомендует `lite`. В `lite`:
  - фаза 6 использует облегчённый путь: один имплементер на весь план + одно финальное
    whole-branch ревью (без per-task fresh-субагента и без per-task review);
  - self-review остаётся, spec-review/plan-review остаются (они дёшевы и ловят баги).
  Оркестратор описывает lite-путь как надстройку над SDD; сам vendored SDD-скилл трогаем
  минимально (одна строка: «если pipeline_profile=lite — см. lite-протокол оркестратора»).
- **Критерий:** state знает profile; оркестратор описывает lite-протокол фазы 6; порог явный.
- **Риск:** касается vendored SDD одной строкой → UPSTREAM.md обновить.
- **state-схема:** добавить поле `pipeline_profile: full | lite | null` в §4 (обновить спеку).

### A4. idea-intake — умный пропуск уже озвученного
- **Файл:** `skills/idea-intake/SKILL.md` (new)
- **Проблема:** 9 вопросов строго по одному тяжело, когда половина ответов уже в первом
  описании идеи.
- **Правка:** разрешить: после первого описания идеи агент выделяет, на какие из 9 пунктов
  ответ уже дан, и предъявляет их владельцу одним сообщением «вот что я понял: … — подтверди
  или поправь», спрашивая по одному ТОЛЬКО реально недостающее. Дисциплина «по одному» сохраняется
  для неясного; экономится онбординг.
- **Критерий:** скилл описывает извлечение-из-описания + подтверждение пачкой + добор недостающего.

### A5. product-discovery — fallback при отсутствии веб-поиска
- **Файл:** `skills/product-discovery/SKILL.md` (new)
- **Проблема:** субагенту в песочнице/headless недоступен веб-поиск; скилл строит quick вокруг
  research-субагента и не описывает поведение без веба — субагент импровизирует.
- **Правка:** добавить явный fallback: если у research-субагента нет веб-инструментов —
  отвечать по знаниям модели, ПОМЕЧАТЬ выводы «не верифицировано против первоисточников»,
  не выдумывать URL; отчёт обязан нести эту пометку. (Согласуется с уже имеющейся оговоркой
  про отсутствие adversarial в quick.)
- **Критерий:** скилл описывает web-unavailable fallback + обязательную пометку в артефакте.

## Группа B — важные уникальные (по одному прогону, но настоящие дефекты)

### B1. mvp-scoping сверяет must-список с дифференциатором/killer из discovery
- **Файл:** `skills/mvp-scoping/SKILL.md` (new)
- **Проблема (прогон B):** discovery назвал единственный дифференциатор, а mvp-scoping не тянет
  этот вердикт в must — дифференциатор легально уехал в later, продукт обессмыслился.
- **Правка:** в Process добавить шаг: прочитать из `01-discovery-report.md` дифференциатор и
  раздел «idea killers»; при приоритизации явно проверить, что дифференциатор попал в must,
  а если уходит в later/never — обосновать это владельцу на гейте продуктовым языком.
- **Критерий:** скилл требует сверки must ↔ дифференциатор discovery.

### B2. plan-writing — типы верификации задачи (unit / acceptance / manual)
- **Файл:** `skills/plan-writing/SKILL.md` (reworked)
- **Проблема (прогон B):** TDD-мандат «падающий юнит-тест на каждую задачу» конфликтует со
  стеком без раннера (DOM-склейка на чистом вебе без jsdom, jsdom нарушает «без зависимостей»).
- **Правка:** разрешить каждой задаче объявлять тип верификации: `unit` (TDD как сейчас),
  `acceptance` (проверяется демо-сценарием, юнит-тест не требуется), `manual` (ручная проверка
  с явным чеклистом). Задача без юнит-теста легальна, если помечена acceptance/manual и это
  обосновано стеком. НЕ ослаблять TDD там, где unit-тест возможен.
- **Критерий:** шаблон задачи в plan-writing поддерживает поле verification; TDD не требуется
  для acceptance/manual задач; обоснование обязательно.

### B3. stack-selection называет, чем гоняются тесты
- **Файл:** `skills/stack-selection/SKILL.md` (new)
- **Проблема (прогон B):** стек «без сборки» скатывается в «тестировать нечем»; конфликт с
  TDD-фазами всплывает поздно (фаза 5/6).
- **Правка:** в артефакт `03-stack.md` добавить обязательный пункт: «чем гоняются тесты в этом
  стеке» (раннер/фреймворк) и какие части покрываются unit vs только acceptance. Критерий 4
  (совместимость с деплоем) дополнить сверкой product_type с деплой-таргетами по имени.
- **Критерий:** скилл требует назвать тест-раннер; связка с B2.

### B4. quick-discovery предупреждает про решение go без данных о спросе
- **Файл:** `skills/product-discovery/SKILL.md` (new)
- **Проблема (прогон B):** в quick market/demand пропущен by design, но гейт go/pivot/stop
  просит бизнес-решение — владельца не предупреждают, что спрос не исследован.
- **Правка:** в quick-отчёте обязательная строка-предупреждение владельцу: «решение go
  принимается БЕЗ данных о спросе; если спрос под вопросом — вернись и запусти deep».
- **Критерий:** quick-ветка требует это предупреждение (рядом с оговоркой про adversarial).

## Группа C — дешёвые баги ядра

### C1. Hook не грузит два оркестратора разом
- **Файлы:** `README.md` (усилить), опц. `hooks/session-start`
- **Проблема (прогон A):** при установленном рядом obra/superpowers SessionStart-хуки ОБОИХ
  плагинов инжектят почти идентичные «EXTREMELY_IMPORTANT» блоки (using-superpowers +
  using-super-puper-powers) — дублирование системного промпта.
- **Правка:** SPP не может отключить чужой хук. Реалистичный фикс — усилить README Compatibility:
  на время SPP-pipeline `superpowers` отключать (уже есть, но сделать это заметнее/жёстче).
  Опционально: в session-start добавить комментарий-детект (если виден инжект using-superpowers —
  это не ломает, но отметить в доке). Реальный технический скип чужого плагина вне возможностей SPP.
- **Критерий:** README чётко предупреждает; ничего не выдумываем про чужой хук.
- **Примечание:** это ограничение среды сосуществования, не баг SPP как такового; правим доку.

### C2. finishing-a-development-branch — машинный чек pipeline-state
- **Файлы:** `skills/subagent-driven-development/SKILL.md` (VENDORED),
  `skills/finishing-a-development-branch/SKILL.md` (VENDORED),
  `skills/using-super-puper-powers/SKILL.md` (reworked, усилить)
- **Проблема (прогон A):** перехват перехода в finishing держится на памяти агента; vendored SDD
  ведёт прямо в finishing, про HARD-GATE оркестратора не зная. При сжатии контекста гейт проскочит.
- **Правка (минимальная для vendored):** в начало `finishing-a-development-branch` добавить
  guard-строку: «Если существует `docs/spp/pipeline-state.md` и `06-acceptance-demo.md` не approved —
  СТОП, вернись к оркестратору super-puper-powers, гейт фазы 6 не пройден». То же — одна строка в
  SDD перед вызовом finishing. Усилить формулировку в оркестраторе.
- **Критерий:** guard присутствует в finishing; UPSTREAM.md отражает модификацию обоих vendored-файлов.
- **Риск:** vendored → шапки Modifications + UPSTREAM.md.

## Группа D — git-деградация точечно (выбор: точечно, не полный dry-run)

### D1. Явная деградация git-шагов
- **Файлы:** `skills/spec-writing/SKILL.md` (reworked), `skills/plan-writing/SKILL.md` (reworked),
  `skills/subagent-driven-development/SKILL.md` (VENDORED), `skills/release-fixation/SKILL.md` (new)
- **Проблема (оба прогона):** git-шаги (commit в spec/plan, per-task commit + finishing в SDD,
  git tag в release) в среде без git-записи упираются в стену; плагин не отличает «шаг провален»
  от «шаг неисполним», толкает в тупик или к нарушению hard-gate о фабрикации.
- **Правка:** там, где скилл предписывает git-операцию, добавить деградацию: «если git-запись
  недоступна в этом окружении — работай по файлам на диске, зафиксируй в артефакте фазы пометку
  „git-шаг не исполнен (окружение без git-записи)“; НЕ фабрикуй результат». Явно НЕ трогаем
  hard-gate про фабрикацию — наоборот, даём легальный путь «не исполнено, помечено».
  Для spec-writing/plan-writing: коммит документов — не блокирующий шаг фазы (перенести в
  release-fixation или сделать best-effort с пометкой).
- **Критерий:** каждый git-шаг имеет явную деградацию + честную пометку; фабрикация по-прежнему запрещена.
- **Риск:** vendored SDD → одна секция про git-деградацию + UPSTREAM.md.

### D2. Под-режим strategy-only для фаз 8–9
- **Файлы:** `skills/deploy-strategy/SKILL.md` (new), `skills/post-release/SKILL.md` (new),
  `skills/using-super-puper-powers/SKILL.md` (reworked, gate-map)
- **Проблема (оба прогона):** фазы 8–9 бинарны — либо реальный прод с evidence, либо гейт
  нечестен. Нет режима «стратегия выбрана, runbook готов, деплой отложен».
- **Правка:** deploy-strategy получает два режима гейта:
  - `deployed` — как сейчас: «продукт живёт по X, прод-смоук пройден — evidence — принять?»;
  - `strategy-only` — «стратегия выбрана, runbook готов, деплой отложен» — БЕЗ требования
    live-evidence; `deploy_target` помечается «выбран, не исполнен»; прод-смоук переносится на
    момент реального деплоя.
  post-release учитывает: если deploy помечен «не исполнен» — памятка в режиме «когда задеплоишь»,
  мониторинг/канал описываются как инструкция к будущей настройке, фаза 9 не считает прод живым.
  Для web-статики без секретов — часть инвариантов явно N/A с обоснованием.
- **Критерий:** deploy-strategy различает два режима гейта; post-release уважает «деплой отложен».
- **state-схема:** значение `deploy_target` допускает пометку статуса (исполнен/отложен) — уточнить в §4.

## Порядок реализации

1. Обновить спеку (§4 state: pipeline_profile, deploy-статус; §5 — новые требования) — фундамент.
2. Новые скиллы (A4, A5, B1, B3, B4) — независимы, параллелятся.
3. Reworked (spec-writing A1/A2/D1, plan-writing A2/A3/B2/D1) — независимы между собой.
4. Оркестратор (A3 lite-протокол, C2, D2 gate-map) + deploy-strategy/post-release (D2) + release-fixation (D1).
5. Vendored-правки (SDD, finishing) + UPSTREAM.md — деликатно, минимально.
6. README (C1), CHANGELOG.
7. Приёмка: grep-инварианты + frontmatter + 3-H2 (как в v0.1 T22); опц. повторный догфуд.

Каждая содержательная правка — с независимым ревью (как в v0.1).
