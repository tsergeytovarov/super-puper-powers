# Super Puper Powers — дизайн pipeline

> v0.2 — обновлено по итогам ревью. Цель: скилл-система, которая доводит идею до задеплоенного продукта силами человека **без навыков разработчика**.

## Принятые решения (ревью v0.1)

1. **Архитектура — C (гибрид/vendored).** Везде явно указываем: репозиторий основан на `obra/superpowers` конкретной версии (см. §5).
2. **Discovery — два режима:** quick и deep.
3. **Юрисдикция** уточняется на старте (в idea-intake).
4. **Деплой** — не привязка к площадкам, а сначала *выбор* стратегии под проект, потом исполнение.
5. **Post-release** включаем в v1.
6. **Язык артефактов** — из настроек пользователя.
7. **Новый скилл `stack-selection`** — выбор стека до фазы дизайна.

---

## 1. Superpowers как есть

**Что это.** 14 композируемых скиллов + hooks, которые заставляют агента следовать методологии автоматически. Философия: hard gates (нельзя писать код до утвержденного дизайна), субагенты с чистым контекстом, TDD, «evidence before claims», ревью после каждой задачи.

**Основной pipeline superpowers:**

| # | Фаза | Скилл | Артефакт | Гейт |
|---|------|-------|----------|------|
| 1 | Дизайн через диалог | `brainstorming` | `docs/superpowers/specs/*-design.md` | self-review спеки + user review спеки |
| 2 | План реализации | `writing-plans` | `docs/superpowers/plans/*.md` — задачи «для джуна без контекста», каждый шаг 2–5 мин с готовым кодом | self-review плана; выбор способа исполнения |
| 3 | Реализация | `subagent-driven-development` (или `executing-plans`) | код + коммиты | ревьюер-субагент после **каждой** задачи (spec compliance + качество), финальное whole-branch ревью |
| 4 | Проверка | `verification-before-completion` | вывод команд как доказательство | «no claims without fresh evidence» |
| 5 | Интеграция | `finishing-a-development-branch` | merge / PR / branch | тесты зелёные + выбор пользователя из 4 опций |

**Поддерживающие скиллы:** `test-driven-development`, `systematic-debugging`, `requesting/receiving-code-review`, `using-git-worktrees`, `dispatching-parallel-agents`, `writing-skills`, `using-superpowers` (диспетчер: «если есть хоть 1% шанс, что скилл применим — обязан вызвать»).

**Сильные механики, которые забираем как есть:** hard gates, чистый контекст субагентов, двухступенчатое ревью (per-task + whole-branch), выбор модели по сложности задачи, статусы субагентов (DONE / DONE_WITH_CONCERNS / NEEDS_CONTEXT / BLOCKED), progress ledger, verification-дисциплина.

## 2. Гэпы superpowers относительно нашей цели

### 2.1. До кода — фаз просто нет
- **Product discovery отсутствует.** Brainstorming сразу лезет в технический дизайн, предполагая, что идея валидна. Нет анализа конкурентов, рынка, юридических рисков, оценки осуществимости.
- **MVP-scoping отсутствует.** YAGNI работает на уровне кода, но никто не решает на продуктовом уровне «что режем из первой версии».
- **Выбор стека отсутствует.** Стек возникает неявно внутри технического дизайна, без явного сравнения вариантов и без учёта того, кто будет это сопровождать.

### 2.2. Спецификации — ревью слабее, чем кажется
Self-review спеки и user review gate в brainstorming **есть**. Но:
- **Нет независимого adversarial-ревью спеки субагентом.** Автор проверяет сам себя «свежим взглядом» — это ловит опечатки и TBD, но не слепые зоны автора. Для кода ревьюер-субагент обязателен, для спеки — нет. Непоследовательно.
- **Нет ревью после декомпозиции.** Если проект разбит на под-проекты, каждый получает свою спеку в своём цикле. Никто не ревьюит набор спецификаций целиком: согласованность интерфейсов, дыры на стыках, противоречия.
- **User review спеки предполагает читателя-разработчика.**

### 2.3. План — только self-review
Плану, по которому субагенты будут работать часами, не назначается независимый ревьюер. Ошибка в плане размножается по всем задачам.

### 2.4. После merge — обрыв
Pipeline заканчивается на merge/PR. Нет: релиза и версионирования, упаковки, деплоя, smoke-тестов на проде, мониторинга, петли обратной связи.

### 2.5. Аудитория
Все гейты сформулированы на языке кода. Не-разработчику нужны гейты на языке продукта: сценарии, демо, скриншоты.

## 3. Целевой pipeline Super Puper Powers

Сквозной принцип: **каждая фаза = вход → работа → артефакт-документ → гейт с человеком на языке продукта**. Порядок фаз 0→9 — рекомендованный маршрут, а не жёсткая последовательность: оркестратор маршрутизирует по контексту запроса (как upstream `using-superpowers`), любой скилл вызывается напрямую, каждая фаза в конце подсказывает следующую и предлагает открыть её в новом чате. Память между чатами хранится в необязательном журнале `docs/spp/pipeline-state.md` — прервать сессию и вернуться можно с любого места; журнал ничего не гейтит. Язык артефактов берётся из настроек пользователя (фиксируется в журнале при старте).

### Фаза 0. Idea Intake — `idea-intake`
- **Вход:** идея в свободной форме.
- **Работа:** структурированный диалог (по одному вопросу): проблема, для кого, чем отличается, критерий успеха, ограничения — бюджет, сроки, **юрисдикция** (обязательный вопрос: где живут пользователи и где живёт автор — определяет юр-анализ в discovery), язык артефактов.
- **Артефакт:** `00-idea-brief.md`.
- **Гейт:** «Я правильно понял идею?»

### Фаза 1. Discovery — `product-discovery`
- **Режимы:** пользователь выбирает на входе:
  - **quick** (~30 мин): один research-субагент, топ-конкуренты, очевидные юр-стопы, грубая оценка осуществимости;
  - **deep** (часы): параллельные субагенты — (a) конкуренты и аналоги, (b) юридические риски под юрисдикцию из brief, (c) рынок и спрос, (d) осуществимость и порядок стоимости эксплуатации — плюс adversarial-проверка фактов.
- **Артефакт:** `01-discovery-report.md` с разделом «Убийцы идеи, если есть» и рекомендацией go / pivot / no-go.
- **Гейт:** решение go / pivot / stop. Единственная фаза, где нормальный исход — «остановиться и сэкономить месяц».

### Фаза 2. MVP Scoping — `mvp-scoping`
- **Работа:** полный список фич из brief + discovery → приоритизация (must / later / never) → walking skeleton: минимальный сквозной сценарий, доказывающий ценность. Метрики успеха MVP. Явный список «чего в MVP НЕ будет».
- **Артефакт:** `02-mvp-scope.md`.
- **Гейт:** утверждение состава MVP — как списка пользовательских сценариев, не фич.

### Фаза 3. Stack Selection — `stack-selection` *(новое, по итогам ревью)*
- **Вход:** MVP-scope + ограничения из brief.
- **Работа:** определить тип продукта (web / пакет / tg-бот / комбинация) и предложить 2–3 варианта стека с trade-offs. Критерии подбора — с поправкой на не-разработчика:
  - **сопровождаемость агентом:** мейнстримный стек с большим корпусом примеров — агент будет писать и чинить его лучше экзотики;
  - **стоимость и простота эксплуатации:** сколько стоит хостинг, есть ли free tier, насколько сложен деплой;
  - **скорость до MVP:** batteries-included фреймворки vs гибкость;
  - **совместимость с будущим деплоем:** стек не должен загонять в дорогую инфраструктуру.
- Рекомендация формулируется на языке последствий для владельца («вариант А: бесплатный хостинг, обновления в одну команду; вариант Б: гибче, но ~$20/мес и сложнее»), а не технических свойств.
- **Артефакт:** `03-stack.md` — выбранный стек + отвергнутые варианты с причинами.
- **Гейт:** утверждение варианта.

### Фаза 4. Design & Spec — `spec-writing` (обёртка над `brainstorming`)
- **Работа:** технический дизайн по методике brainstorming, но вход — утвержденные MVP-scope и стек, поэтому вопросы к человеку только про продуктовые решения (UX, поведение); технические решения агент принимает сам и фиксирует с обоснованием. Декомпозиция на под-проекты при необходимости.
- **Новое vs superpowers:**
  - **`spec-review` — независимый adversarial-ревьюер спеки** (субагент, чистый контекст): полнота относительно MVP-scope, противоречия, неоднозначности, нереализуемость.
  - **`cross-spec-review` — ревью набора спецификаций после декомпозиции:** согласованность интерфейсов, дыры на стыках, порядок сборки.
- **Артефакты:** `04-specs/*.md` + резюме для гейта на языке продукта: сценарии «пользователь делает X → происходит Y», эскизы экранов.
- **Гейт:** человек утверждает продуктовое резюме, не архитектуру.

### Фаза 5. Planning — `plan-writing` (обёртка над `writing-plans`)
- **Работа:** как в superpowers, плюс **`plan-review` — независимый ревьюер плана** (покрытие спеки, типовая согласованность между задачами, отсутствие плейсхолдеров).
- **Артефакт:** `05-plans/*.md`.
- **Гейт:** лёгкий — «спека покрыта планом на N задач, оценка объёма такая-то, стартуем?».

### Фаза 6. Implementation — `subagent-driven-development` (vendored, без изменений)
- TDD, per-task ревью, whole-branch ревью, worktree-изоляция.
- **Гейт:** не diff, а **демо**: агент поднимает локальный preview / собирает пакет / запускает бота в тестовом режиме и проводит человека по сценариям из MVP-scope. Acceptance = «каждый сценарий работает у меня на глазах».
- Перед демо — два оркестраторских чекпоинта: `data-boundaries`, `pre-show-audit`. Плюс шесть отдельных helper-скиллов на вызов по ситуации. Детали и источник истины — в спеках скиллов, не здесь.

### Фаза 7. Acceptance & Fixation — `release-fixation`
- **Работа:** `verification-before-completion` + `finishing-a-development-branch` + версия (semver), changelog на языке пользователя, тег.
- **Артефакт:** `06-release-notes.md`.
- **Гейт:** «фиксируем версию X?»

### Фаза 8. Deploy — `deploy-strategy` + исполнение *(переработано по итогам ревью)*
Главная ценность скилла — **выбрать, как лучше деплоить именно этот проект**, а не исполнить зашитый рецепт. Два шага:

**Шаг 1 — выбор стратегии (`deploy-strategy`):**
- собрать вводные: тип продукта, стек (из `03-stack.md`), бюджет, есть ли у пользователя существующие аккаунты/инфраструктура (у всех разная — ничего не предполагаем), требования юрисдикции к размещению данных;
- предложить 2–3 варианта с trade-offs на языке владельца: стоимость/мес, сложность обновления, привязка к вендору, что случится при росте нагрузки;
- **гейт:** пользователь выбирает вариант.

**Шаг 2 — исполнение:**
- reference-плейбуки как справочный материал (не жёсткий рецепт): web-приложения, пакеты/плагины (npm / PyPI / marketplace), tg-боты (BotFather, long polling vs webhook);
- общие инварианты: секреты никогда не в git; деплой повторяем (скрипт в репо); smoke-тест на проде обязателен — verification-дисциплина распространяется на прод.
- **Артефакт:** `07-deploy-runbook.md` — как задеплоено, как обновлять, как откатить, сколько стоит.
- **Гейт:** «продукт доступен по адресу X, сценарии на проде проверены — принимаешь?»

### Фаза 9. Post-Release — `post-release` *(в v1)*
- Минимальный мониторинг (uptime, ошибки) в рамках выбранной стратегии деплоя, канал обратной связи, петля: фидбек → новый idea brief → цикл заново с фазы 0 или 2.
- **Артефакт:** `08-operations.md` — что мониторится, куда смотреть, что делать при инциденте.

### Карта: своё vs superpowers

| Фаза | Скилл | Происхождение |
|------|-------|---------------|
| 0–3 | idea-intake, product-discovery, mvp-scoping, stack-selection | **новое** |
| 4 | spec-writing + spec-review + cross-spec-review | обёртка над brainstorming + **новые ревью** |
| 5 | plan-writing + plan-review | обёртка над writing-plans + **новое ревью** |
| 6 | subagent-driven-development, TDD, code review | **vendored superpowers** |
| 7 | release-fixation | обёртка над finishing-a-development-branch |
| 8 | deploy-strategy + плейбуки | **новое** |
| 9 | post-release | **новое** |
| — | spp-pipeline (оркестратор-диспетчер + журнал) | **новое**, аналог using-superpowers |

## 4. Принципы для не-разработчика

1. **Гейты на языке продукта.** Человек утверждает сценарии, демо, скриншоты и последствия для кошелька — никогда не diff и не архитектуру.
2. **Технические решения агент принимает сам** и фиксирует с обоснованием. У человека спрашивают только то, на что он реально может ответить.
3. **Resumability.** Необязательный журнал `pipeline-state.md` — это память между чатами: где остановились, какие решения приняты, язык артефактов. Сессия читает его для контекста, а не как enforcing-стейт: он ничего не блокирует и не навязывает следующую фазу.
4. **Каждая фаза оставляет документ.** Через полгода можно понять, почему продукт такой.
5. **Право на no-go.** Discovery легитимно завершает проект — это успех pipeline, а не провал.

## 5. Архитектура: решение — C (vendored-гибрид)

- Vendored-копии скиллов реализации: `subagent-driven-development`, `test-driven-development`, `verification-before-completion`, `finishing-a-development-branch`, `using-git-worktrees`, `requesting/receiving-code-review`, `systematic-debugging`, `dispatching-parallel-agents`. Обёртки/переработки: `brainstorming` → `spec-writing`, `writing-plans` → `plan-writing`.
- **Атрибуция — обязательна и явна:**
  - в README: «Основано на [obra/superpowers](https://github.com/obra/superpowers) v<X.Y.Z> (MIT), автор Jesse Vincent»;
  - в шапке каждого vendored/переработанного SKILL.md: `> Vendored from obra/superpowers v<X.Y.Z>, commit <sha>. Modifications: <список или "none">`;
  - файл `UPSTREAM.md`: версия-источник, список файлов, дата последнего sync, процедура обновления;
  - лицензия MIT сохраняется, копия оригинального LICENSE в репо.
- Периодический ручной sync с апстримом по процедуре из `UPSTREAM.md`.

## 6. Что дальше

Спека утверждена по составу фаз и архитектуре. Следующий шаг — применить pipeline к самому себе: план реализации плагина (структура репо, порядок написания скиллов: оркестратор → фазы 0–3 → обёртки 4–5 → vendoring 6–7 → деплой 8–9), затем сборка.
