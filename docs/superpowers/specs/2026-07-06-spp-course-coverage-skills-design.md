# SPP — скилы покрытия курса (сабпроект 1)

> **Спека для реализации.** Самодостаточна: контекст обсуждения не требуется.
> Дизайн-обоснование и карта покрытия — в этом же файле, разделы 1-3.
>
> **Для агента-исполнителя:** это спека, а не план. Следующий шаг — implementation
> plan (через `writing-plans`), затем реализация. Все правки — в репозитории SPP
> (`super-puper-powers`), курс (`apps/course`) в этот заход не трогаем — это
> отдельный сабпроект 2, зависящий от этого.

## 1. Цель

Курс «От идеи до продакшена» (`apps/course`) учит человека без навыков разработчика
пройти путь идея → MVP. Его хребет — это ровно пайплайн SPP: 16 модулей почти 1:1
ложатся на фазы `idea-intake` … `post-release`. Но два модуля хребта и часть
сайд-квестов курса SPP пока не покрывает. Цель сабпроекта 1 — закрыть эти дыры
восемью новыми скилами, чтобы курс мог целиком опереться на SPP, а не на upstream
`obra/superpowers` плюс ручные заглушки.

Ключевые принципы (нарушение любого = дефект):

1. Новые фазовые проверки не ломают проверенную машину состояний: нумерация фаз
   0-9 и артефактов `docs/spp/0N-*.md` остаётся прежней, hard-gate на
   `06-acceptance-demo.md` — целым.
2. Гейты с человеком остаются только на языке продукта. Новые проверки —
   агентские, их находки становятся задачами на фикс, а не гейтом для
   ученика-неразработчика.
3. Helper-скилы стоят вне state machine и не связаны с `pipeline-state.md`.
4. Все восемь скилов — SPP-original, не vendored из upstream.
5. Источник методологии — уже написанный контент курса; скил дистиллирует
   *действие агента*, а не пересказывает педагогику.

## 2. Карта покрытия (обоснование)

Весь контент курса (16 модулей + 22 сайд-квеста) против набора скилов SPP делится
на четыре корзины:

- **Корзина 1 — хребет, покрыт фазовыми скилами.** Модули 1-3, 6-12, 15-16 →
  `idea-intake`, `product-discovery`, `mvp-scoping`, `brainstorming`,
  `stack-selection`, `spec-*`, `plan-*`, `subagent-driven-development`, `TDD`,
  `verification-before-completion`, `finishing-a-development-branch`,
  `release-fixation`, `deploy-strategy`, `post-release`. Дыр нет.
- **Корзина 2 — реальные дыры в пайплайне.** Модуль 13 «Где живут данные» и
  модуль 14 «Проверки перед показом» + минимальная безопасность. Нет скилов.
- **Корзина 3 — сайд-квесты, годные в standalone-скилы.** Отобраны шесть:
  `accessibility`, `mobile-version`, `test-runner-setup`, `seo-baseline`,
  `geo-optimization`, `ux-copywriting`.
- **Корзина 4 — чистая педагогика, скилами быть не должна.** `glossary`,
  `golden-artifacts`, `ii-bez-terminatora`, `ai-rules`, `where-to-work`,
  `ai-interviewer`, `orchestration`, `tdd-human` и остаток корзины 3
  (`token-optimization`, `choose-model`, `agent-settings-git`, `pages-and-links`,
  `hosting-domains`, `auth-when-needed`, `minimal-analytics`). Не трогаем.

## 4. Фазовые скилы: размещение (вариант B)

### 4.1. Куда встают

Обе проверки — orchestrator-owned шаги в завершении фазы 6, ровно как уже устроен
acceptance demo. Новых номеров фаз не добавляем. Последовательность закрытия фазы 6:

```
subagent-driven-development (сборка + финальное whole-branch ревью)
  → data-boundaries      (проверка: где живут данные, экспорт, границы)
  → pre-show-audit       (проверка: скрытые риски перед показом + безопасность)
  → acceptance demo      (гейт фазы 6, без изменений)
  → release-fixation (фаза 7)
```

Порядок повторяет курс: m13 → m14 → m15 (DoD/приёмка). Обе проверки идут **до**
acceptance demo — демонстрируем уже вычищенный продукт.

### 4.2. Гейты и артефакты

Обе проверки — агентские, **своего человеческого гейта не имеют**. Их находки
становятся задачами на фикс; единственный человеческий гейт фазы 6 остаётся
acceptance demo. Это сохраняет правило «гейт только на языке продукта»: аудит-отчёт
не показываем ученику как точку принятия решения.

Артефакты кладём рядом с `06-acceptance-demo.md`, без переномеровки:

- `docs/spp/06-data-boundaries.md`
- `docs/spp/06-pre-show-audit.md`

### 4.3. Правки машины состояний

Оркестратор `using-super-puper-powers`, раздел **Phase 6 Gate Ownership**:

1. После финального whole-branch ревью и **до** acceptance demo оркестратор
   последовательно запускает `data-boundaries`, затем `pre-show-audit`.
2. Каждая проверка пишет свой артефакт; блокеры из находок становятся задачами на
   фикс и чинятся до acceptance demo (как уже устроено с провалом сценария демо).
3. В `pipeline-state.md` фиксируем факт прогона: поля
   `data_boundaries_checked: true|false` и `pre_show_audit_checked: true|false` с
   путями артефактов. Это защита от compacted-context: перед acceptance demo
   оркестратор машинно проверяет, что оба поля `true`, а не полагается на память.
4. Hard-gate на `06-acceptance-demo.md` не меняется. Добавляем спутниковое правило:
   acceptance demo не запускается, пока оба чекпоинта не отмечены в state.

Нумерация фаз 0-9, имена downstream-артефактов (`07-release-notes.md` и далее) и
существующие hard-gates остаются нетронутыми.

### 4.4. Скил `data-boundaries`

- **Класс:** фазовый чекпоинт, orchestrator-owned.
- **Вход:** собранный продукт + storage-решение из `03-stack.md` / `04-specs/`.
- **Действие:** сделать хранение явным (где физически живут данные), проверить
  корректность export, очертить границы хранения и риски, назвать следующий слой
  роста. Если storage в стеке/спеке остался неявным — зафиксировать выбор здесь.
- **Выход:** `docs/spp/06-data-boundaries.md` — карта хранения + список граничных
  находок (риск → блокер? → фикс/отложить).
- **Источник:** `apps/course` модуль 13 (`concept.md`, `practice.md`) и квест
  `data-boundaries-sidequest` (s7).
- **Критерий готовности:** storage-решение явно записано; export проверен;
  граничные риски перечислены, каждый помечен как исправленный или отложенный
  с причиной.

### 4.5. Скил `pre-show-audit`

- **Класс:** фазовый чекпоинт, orchestrator-owned.
- **Вход:** собранный продукт после `data-boundaries`.
- **Действие:** аудит скрытых рисков перед первым показом — странный title,
  сломанный mobile, неработающий build, случайный debug-текст, лишние данные в
  export JSON, поле ввода, ломающее интерфейс, — плюс минимальная безопасность
  (утечка данных, обработка ввода). Чинит только то, что блокирует нормальный
  первый показ.
- **Выход:** `docs/spp/06-pre-show-audit.md` — audit report (риск → серьёзность →
  блокер? → фикс).
- **Источник:** `apps/course` модуль 14 (`concept.md`, `practice.md`).
- **Критерий готовности:** audit report собран; все блокеры исправлены либо явно
  отложены с причиной.

## 5. Helper-скилы (6)

Общие свойства: standalone, вне state machine, без связки с `pipeline-state.md`.
Вызываются по требованию — обычно во время фазы 6 (реализация), а `seo-baseline` и
`geo-optimization` также уместны у фазы 8 (deploy). Каждый несёт standalone-триггер
в описании SKILL.md (как фазовые скилы SPP). В оркестратор
`using-super-puper-powers` добавляем короткий раздел **On-demand helpers** со
списком — чтобы агент знал об их существовании и не изобретал ручные ритуалы.

Ни один helper не расширяет scope продукта: это design-only / упаковочные /
проверочные действия над тем, что уже собрано.

### 5.1. `accessibility`
- **Действие:** базовый a11y-аудит: навигация с клавиатуры, контраст, видимый
  фокус, подписи полей, понятность ошибок, базовый screen-reader.
- **Выход:** находки + применённые/перечисленные фиксы.
- **Источник:** квест `accessibility` (s14). **Триггер:** «проверь доступность / a11y».

### 5.2. `mobile-version`
- **Действие:** design-only responsive-проход: existing desktop-UI не разваливается
  на узком экране (overflow, тап-таргеты, формы). Data model и функции не трогаем.
- **Выход:** UI, живущий на мобилке, без расширения scope.
- **Источник:** квест `mobile-version` (s22). **Триггер:** «сделай мобильную версию /
  проверь на телефоне».

### 5.3. `test-runner-setup`
- **Действие:** поставить test runner + один smoke-тест, чтобы «проверка поведения»
  стала командой. Не покрытие всего тестами — рабочий runner и один тест.
- **Выход:** настроенный runner + smoke-тест, проверяемо командой.
- **Источник:** квест `test-runner-setup` (s25). Кормит фазу TDD. **Триггер:**
  «поставь тесты / настрой test runner».

### 5.4. `seo-baseline`
- **Действие:** техупаковка страницы: title, description, favicon, OG-превью,
  robots, sitemap, нормальная структура.
- **Выход:** страница, которой не стыдно делиться ссылкой.
- **Источник:** квест `seo-baseline` (s20). **Триггер:** «сделай SEO baseline».

### 5.5. `geo-optimization`
- **Действие:** упаковка публичной страницы так, чтобы ИИ-ассистенты честно
  понимали продукт: что делает, для кого, ограничения, где факты. Sibling
  `seo-baseline`, техупаковку для поисковиков не дублирует.
- **Выход:** страница, понятная summary-инструментам и ассистентам.
- **Источник:** квест `geo-optimization` (s21). **Триггер:** «GEO / чтобы ИИ понял
  продукт».

### 5.6. `ux-copywriting`
- **Действие:** UI-тексты по контексту экрана, состоянию и действию пользователя;
  микрокопи и состояния (пусто/ошибка/загрузка). Прецедент — `design:ux-copy`.
- **Выход:** тексты интерфейса под текущий сценарий.
- **Источник:** квест `ux-copywriting` (s13). **Триггер:** «напиши UX-тексты /
  микрокопи».

## 6. Атрибуция

(not from upstream)» с перечнем восьми путей. Заголовок-атрибуции obra в их
SKILL.md не нужен — он только для vendored-файлов. `LICENSE.superpowers` не
затрагивается.

## 8. Hooks

Предположительно не трогаем: скилы дискаверятся штатно, чекпоинты гонит оркестратор.
Проверить на этапе плана; если SessionStart-хук или дискавери требуют явного
списка скилов — обновить. Не утверждается как факт до проверки.

## 9. Верификация

У SPP нет JS-тестов. Верификация:

1. Все восемь SKILL.md валидны (frontmatter `name`/`description`, триггеры на месте).
2. Оркестратор видит новые скилы; раздел On-demand helpers перечисляет шесть.
3. Сухой прогон машины состояний фазы 6: после whole-branch ревью оркестратор
   запускает `data-boundaries` → `pre-show-audit` → acceptance demo; поля
   `*_checked` в `pipeline-state.md` выставляются; acceptance demo не стартует без них.
4. `UPSTREAM.md` и оба README согласованы со счётчиком и картой.

## 10. Вне scope

- Курс (`apps/course`) — сабпроект 2, отдельный цикл спека → план → реализация,
  зависит от этого.
- Скилы из `auth-when-needed`, `minimal-analytics` и остатка корзины 3-4 не делаем.
- Переномеровка фаз/артефактов (вариант A) отклонена.
